main.R

The main.R file contains a basic working example of the CSM package. It is easy to use as a starting point for development on the package, as well as for learning about how R handles CSM. For example, the following configuration will execute the aei, aecnt, and aegap analysis - assuming these analyses are properly configured in the excel file, specified by path_csm_excel.

Flagging

A block for an analysis is created by splitting the Flagging Tab on Category and Analysis. From that split, data will be analyzed by applying a flagging criteria to each of the signals within that collection.

For example, split data on Category and Analysis, get sections AE + AEI, AE + AECNT, etc.

Within the AE + AEI category, select the column containing the signal (paramcd) and apply the flagging specification to each signal.

Consider the following two tables. The first table is the number of signals, to a given category and analysis. Each signal has one and only one block of R code that will execute. This is determined by requiring the Flagging Specification to be written within the same chunk as Category and Analysis.

signal_count <-

csm_analysis

The csm_analysis function contains all the logic to call an analysis module. This section “preps” the csm_analysis_loop function. csm_analysis_loop contains a switch for analysis specific code.

If there is no analysis_filter argument, it is assumed that all analyses within the configuration are to be run. analysis_filter is convenient when you want to run a specific analysis repeatedly, like during development.

We loop over csm_analysis_loop with the names of the analyses that are to be run. analysis_names is analysis_filter in the study_signal_generator function.

# csm_analysis is called from study_signal_generator
# after creating the mastersheet
csm_analysis(
  mastersheet = mastersheet, 
  analysis_filter = 'aei'
)

csm_analysis <- 
  function(mastersheet = NULL,
           debugging = FALSE,
           analysis_filter = NULL) {

  analysis_names <- unique(mastersheet$flagging_setup$analysis)

  if (!is.null(analysis_filter)) {
    analysis_names <- analysis_names[
      analysis_names %in% analysis_filter
    ]
  }

  analysis_results <- map(
    analysis_names,
    csm_analysis_loop,
    mastersheet = mastersheet,
    debugging = debugging
  )

  names(analysis_results) <- analysis_names

  analysis_results <- list_to_lower(analysis_results)

  analysis_results
}

Within csm_analysis_loop (reference the file for the full function) you will see a series of switch statements, which correspond to a particular analysis.

ANALYSIS_PREPROCESS takes the mastersheet, and filters for only the data required from the excel sheet to execute the analysis in question, which in this case is aei. Any environment variables created by program configuration parameter are temporarily injected into the environment while the analysis runs. Once the tryCatch is finished processing, the environment variables are reset to their original values. This allows for environment variables with the same name to be different between analyses, for example - aei and misseddose each use CompareProportion.

Apart from the Excel file, if adding a module, this is the first section that needs an update. When adding a new CompareProportion analysis, all that is needed is to copy this section of code and replace “aei” with the new analysis name, update ANALYSIS_PREPROCESS with preferred function names, and generate an analysis_* function that takes the data and subsetted configuration file.

"aei" = tryCatch(expr = {
      {
        input <- ANALYSIS_PREPROCESS(
          mastersheet,
          "aei",
          "compareproportion",
          debugging
        )
        parameters <- input$configuration$parameters
        walk2(
          parameters$parameter,
          parameters$value,
          ~ setenv(..1, ..2, quiet = FALSE)
        )

        on.exit({
          walk(
            parameters$parameter,
            ~ setenv(..1, "", quiet = FALSE)
          )
        })
        result <- analysis_aei(
          input$data,
          input$configuration
        )
      }
    }, error = function(err) {
      tibble(
        error = as.character(err),
        analysis = "aei"
      )
    }),

analysis_aei is an analysis specific wrapper for compare_f . So if you take a look at analysis_aei vs analysis_dose, you will notice they are very similar but slightly different - primarily on the basis of what their r and n values are called. This allows for similar, but different configurations based on which analysis is being run - without needing to change compare_f.
